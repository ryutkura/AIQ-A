# `docker run` コマンドの完全解剖とオプションの順番

`docker run` は非常によく使うコマンドですが、オプションが多くて混乱しやすいですよね。
今回は以下のコマンドを例に、「書く順番のルール」と「各パーツの意味（特に `-v`）」を徹底解説します。

**お題のコマンド:**
```[[bash]]
docker run -p 8888:8888 -v $(pwd):/home/jovyan/work --name myjupyter jupyter/datascience-notebook
```

---

## 1. 最重要ルール：「オプション」と「イメージ名」の順番

多くの人が迷うのが「`-p` とか `-v` はどこに書くの？イメージ名より前？後ろ？」という点です。
[[Docker]]コマンドの絶対的なルールは以下の通りです。

```text
docker run  [① オプションたち]  [② イメージ名]  [③ [[コンテナ]]の中で実行するコマンド(省略可)]
```

*   **なぜこの順番なのか？**
    *   **① オプション**：[[Docker]]というシステム自体への指示です。（「ポートを開けて」「名前をつけて」など）
    *   **② イメージ名**：ここから先は「[[コンテナ]]の中身」に関する話に切り替わります。
    *   **③ コマンド**（今回は省略されています）：[[コンテナ]]の中で最初に実行するコマンドです。

もし `docker run jupyter... -p 8888:8888` のように**イメージ名の後ろ**にオプションを書いてしまうと、[[Docker]]は「`-p`」を[[Docker]]へのオプションではなく、「**[[コンテナ]]の中で実行するコマンドの一部だ**」と勘違いしてしまい、エラーになります。

**結論：オプション（`-p`, `-v`, `--name` 等）は必ず「イメージ名」の前に書く！**

---

## 2. コマンドの各パーツを一つずつ分解

お題のコマンドを順番に見ていきましょう。

1.  **`docker run`**
    *   新しく[[コンテナ]]を作って（create）、起動する（start）コマンドです。
2.  **`-p 8888:8888`** (Publish)
    *   **ポートの接続**です。「パソコン側の8888番ポート」と「[[コンテナ]]側の8888番ポート」を繋ぎます。
    *   これがないと、ブラウザで `localhost:8888` にアクセスしても[[コンテナ]]内のJupyterに繋がりません。
3.  **`-v $(pwd):/home/jovyan/work`** (Volume)
    *   **フォルダの共有（マウント）**です。※後ほど超重点的に解説します！
4.  **`--name myjupyter`**
    *   **[[コンテナ]]の名前**です。これをつけると、後で `docker start myjupyter` や `docker exec myjupyter ...` のように簡単に操作できるようになります（前回のお話ですね！）。
5.  **`jupyter/datascience-notebook`**
    *   **元となるイメージ名**です。ここには[[データサイエンス]]用の環境（[[Python]], Jupyter, Pandas等）が最初から全て詰まっています。

## 3. 重点解説：`-v` (ボリュームマウント) とは何か？

一番ややこしい `-v`（ボリューム）について深く掘り下げます。

書き方の基本は **`-v [自分のパソコンのフォルダ] : [コンテナの中のフォルダ]`** です。

### なぜこれが必要なのか？（[[Docker]]の弱点）
[[コンテナ]]は「使い捨て」を前提としています。[[コンテナ]]を削除すると、**[[コンテナ]]の中で作ったファイル（Jupyterで書いたコードなど）も全て一緒に消滅してしまいます。**
それを防ぐために使うのが `-v` オプションです。

### `-v` がやっていること＝「どこでもドア」
`-v` を使うと、パソコン側（[[Windows]]/Mac）のフォルダと、[[コンテナ]]の中のフォルダを**「四次元ポケットのように直結（同期）」**させることができます。

お題の `-v $(pwd):/home/jovyan/work` を分解すると：
*   **左側 `$(pwd)`**：パソコン側のフォルダ
    *   `pwd` は Print Working Directory の略で、ターミナルで**「今現在いるフォルダ」**のパスに置き換わる魔法のコマンドです。
*   **右側 `/home/jovyan/work`**：[[コンテナ]]側のフォルダ
    *   Jupyterイメージの中で、作業用に用意されているデフォルトの場所です。

### つまりどういうこと？
「**今ターミナルを開いている手元のフォルダ**」と、「**[[コンテナ]]の中の `/home/jovyan/work`**」をリアルタイムで同期させます。

*   Jupyter上でコードを書いて保存すると、**手元のパソコンのフォルダにもそのファイルが直接保存**されます。
*   逆に、手元のパソコンのフォルダにデータセット（CSVファイルなど）をポイッと入れると、**Jupyter上からもそのデータが見える**ようになります。
*   もし[[コンテナ]]を間違って消してしまっても、**手元のパソコンにファイルが残っているので絶対に安全**です。

これが `-v` オプションの最大の魅力であり、必須級のオプションである理由です！

## 4. 実践Q&A：「[[コンテナ]]側のパス」ってどうやって調べるの？

**Q. [[コンテナ]]側の作業場所（今回の `/home/jovyan/work` など）を知らなかったら詰みじゃないですか？ドキュメントがない場合はどう探すの？**

非常に鋭い実践的な視点です！おっしゃる通り、ドキュメントが不十分なイメージを使う場合、どこをマウントすればいいのか分からず立ち往生することは実務でもよくあります。
世の開発者たちは、ドキュメントがない「謎のイメージ」に遭遇した時、以下のような方法で内部構造を探り当てています。

### 方法1：とりあえず中に入って探検する（一番よくやる手法！）
ドキュメントがないなら、直接入って確認するのが一番手っ取り早く確実です。以下のように `[[bash]]` や `sh` を指定して[[コンテナ]]を起動し、中に入り込みます。
```[[bash]]
docker run -it --rm [謎のイメージ名] /bin/[[bash]]
```
中に入ったら、`pwd` コマンドで「自分が最初に落とされた場所（デフォルトの作業ディレクトリ）」を確認したり、`ls` で重要なファイルやデータがどこにあるかを探検して特定します。特定できたら `exit` で抜け、本番の `run` コマンドを組み立てます。

### 方法2：`docker image inspect` で設定を覗き見する
イメージの中身を実行せずに解析するコマンドです。
```[[bash]]
docker image inspect [イメージ名]
```
すると大量の設定情報が出てきますが、その中の **`"WorkingDir"`** という項目を見れば、作成者が意図したデフォルトの作業場所が分かります。また **`"Volumes"`** という項目にマウントすべき推奨パスが書かれていることもあります。

### 方法3：元となる `[[Docker]]file` を直接読む
もしそのイメージの[[設計]]図である `[[Docker]]file` を見ることができるなら、中身を直接読みます。
ファイルの中に `WORKDIR /app` のような記述があれば「ここがメインの作業場所だな」と分かります。

公式の有名なイメージ（Jupyterなど）はドキュメント（[[Docker]] HubのREADMEなど）にしっかり書いてありますが、実務でマイナーなイメージや他人が作ったイメージを扱う時は、**「とりあえず `run -it [[bash]]` で突撃して中を覗く」** というサバイバル術が必須スキルになっています！

## 5. さらに深掘り：「[[コンテナ]]起動しちゃったら遅くない？」という疑問

### Q. 方法1の「中に入って探検する」って、その時点でもう[[コンテナ]]作って起動しちゃってるから『時すでに遅し』では…？

**A. 実は全く遅くありません！ここが[[Docker]]最大のパラダイムシフトである「使い捨て（Disposable）」という概念です。**

通常のパソコンや仮想マシン（VM）の感覚だと、「環境を作る＝重い処理で、失敗したらゴミが残るから後戻りしにくい」と感じるのが普通です。あなたのその感覚は、従来のIT知識としては全く正しく、非常に鋭いです。

しかし、[[Docker]]の[[コンテナ]]は**「シャボン玉のように一瞬で作れて、一瞬で消せる」**ほど超軽量な作りになっています。
そして、先ほどの探検コマンドにこっそり忍ばせていた **`--rm`** というオプションが最大のポイントです。

```[[bash]]
docker run -it --rm [イメージ名] /bin/[[bash]]
```

この `--rm` は「**ターミナルから抜けたら（exitしたら）、即座にこの[[コンテナ]]を跡形もなく完全消去してね**」という魔法のオプションです。

つまり、実際には以下のような流れで作業しています：
1. `run -it --rm` で「**偵察用の使い捨て[[コンテナ]]**」を1秒でポーンと作る。
2. 中を覗いて、マウントすべきパス（`/home/jovyan/work` など）のメモを取る。
3. `exit` と打って抜ける。（この瞬間、偵察用[[コンテナ]]は綺麗さっぱり消滅する）
4. メモしたパスを使って、本番用の本当にやりたかったコマンド（`-v` や `-p` などをフル装備した `run`）を実行する。

[[Docker]]の世界では「とりあえず偵察用の[[コンテナ]]を立てて、確認したら捨てる」というのは、**裏紙に試し書きして丸めて捨てるくらい日常的で軽いアクション**なのです。この「使い捨て感覚」が身につくと、[[Docker]]の恐さがなくなり一気に使いこなせるようになります！
