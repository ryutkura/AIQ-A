# Linuxのプロセスとジョブ管理

ユーザーが「プロジェクト周り」と認識していた概念は、おそらくLinuxにおける**「[[プロセス]] (Process)」**および**「[[ジョブ]] (Job)」**の管理についての概念だと思われます。`kill`コマンドなどが登場するのはまさにこの領域です。

以下に、Linuxにおけるプロセスとジョブの包括的な概要と、主要なコマンドの詳細を解説します。

## 1. プロセス (Process) とは？
**プロセス**とは、「現在実行中のプログラム」のことです。
Linuxでは、コマンドを実行したりアプリを立ち上げたりするたびに、OSがそれを「プロセス」として管理します。
* **PID (プロセスID)**: 各プロセスに割り当てられる一意の識別番号です。OSはこの番号を使ってプロセスを管理・操作します。
* プロセスは階層構造を持っており、親プロセスから子プロセスが生成されます。

## 2. ジョブ (Job) とは？
**ジョブ**とは、シェル（ターミナルでコマンドを受け付けるプログラム）が管理する作業の単位です。
* パイプ（`|`）で繋いだ複数のコマンド（例: `ls -l | grep txt`）は、全体で1つの「ジョブ」として扱われます。
* **ジョブ番号**: シェルがジョブを管理するために割り当てる番号です（`[1]`、`[2]` のように表示されます）。

## 3. フォアグラウンドとバックグラウンド
プロセス（ジョブ）の実行状態には大きく分けて2つのモードがあります。

* **フォアグラウンド (Foreground)**: ターミナルの画面を占有して実行される状態。実行中は他のコマンドを入力できません。
* **バックグラウンド (Background)**: ターミナルの裏側で実行される状態。実行中であってもターミナルは解放され、別のコマンドを入力できます。

## 4. 主要なコマンドとショートカット

### プロセスの確認
* **`ps`**: 現在実行中のプロセス一覧を表示します。
  * `ps aux` や `ps -ef` といったオプションをつけて、システム全体の全プロセスを詳細に確認することが多いです。
* **`top` / `htop`**: プロセスの実行状況（CPUやメモリの使用率）をリアルタイムで表示します。

### ジョブの管理
* **`jobs`**: 現在のシェルで管理しているバックグラウンドや一時停止中のジョブ一覧を表示します。
* **`コマンド &`**: コマンドの末尾に `&` をつけると、そのコマンドを最初からバックグラウンドで実行します。
  * 例: `sleep 100 &`
* **`Ctrl + Z`**: フォアグラウンドで実行中のジョブを**一時停止 (Suspend)** し、バックグラウンドに回します。
* **`bg`**: 一時停止中のジョブをバックグラウンドで実行再開させます。
* **`fg`**: バックグラウンドのジョブや一時停止中のジョブを、フォアグラウンドに呼び戻します（例: `fg %1` でジョブ1番をフォアグラウンドへ）。

### プロセスの終了・シグナル送信
* **`Ctrl + C`**: フォアグラウンドで実行中のプロセスを**強制終了**させます（正しくは「割り込みシグナル」を送る）。
* **`kill`**: プロセスやジョブに対して「シグナル」を送るコマンドです。最もよく使われるのがプロセスの終了です。
  * **`kill <PID>`**: 指定したPIDのプロセスに終了要求（SIGTERM）を送ります。プログラムは終了処理を行ってから終わります。
  * **`kill -9 <PID>`**: プロセスを**強制終了 (SIGKILL)** します。通常の`kill`で終了しない暴走プロセスに対して使いますが、データの保存処理などもスキップされるため最終手段です。
  * ジョブに対して使用する場合は、`kill %1` のように `%` とジョブ番号を指定します。

## 5. まとめ
* **プロセス** = OSが管理する「実行中のプログラム」（PIDで管理）
* **ジョブ** = シェルが管理する「ひとまとまりの作業」（ジョブ番号で管理）
* `kill`コマンドは、プロセスを「殺す」というより、プロセスに対して「指示（シグナル）を送る」ためのコマンドであり、その代表的な使い道が「終了させること」です。

## 6. シェル (Shell) と コマンド (Command) の根本的な違い
「人間と機械を繋ぐ」という役割は共通していますが、コンピュータの内部では全く別の実体として動いています。さらに一歩踏み込んで、技術的に何が違うのかを解説します。

### 1. 「常駐する環境」か「使い捨ての道具」か
* **シェル (例: bash, zsh)**:
  ターミナルを開いている間、**常に動き続けている1つのプログラム（プロセス）**です。
  あなたが文字を打ち込めるように待機し、変数（`$USER`など）を記憶したり、`if`文や`for`ループなどのプログラミング的な構文を理解する「環境」そのものです。
* **コマンド (例: ls, cat, grep, kill)**:
  呼び出された時だけ動く**個別のプログラム**です。
  実はコマンドの正体は、OSのディスク上（`/bin/` や `/usr/bin/` などのフォルダ）に保存されている「1個の実行ファイル（アプリ）」に過ぎません。

### 2. コマンドを実行した裏側で起きていること
ターミナルで `ls -l` と入力してEnterを押したとき、シェルとコマンドは以下のように連携しています。

1. **解釈 (シェル)**: シェルが入力された `ls -l` という文字列を読み取り、「`ls` というプログラムに `-l` というオプションを渡せばいいんだな」と解釈します。
2. **探索 (シェル)**: シェルはシステムの中（あらかじめ決められたフォルダ群）から、`ls` という名前の実行ファイルを探し出します（大抵は `/bin/ls` にあります）。
3. **生成 (OS/カーネル)**: シェルはOSの核（カーネル）に対して「この `/bin/ls` というアプリを起動して」と依頼し、OSが新しく**プロセス**を作ります。
4. **実行 (コマンド)**: `/bin/ls` というコマンド（プログラム）がプロセスとして起動し、ファイル一覧を画面に出力します。
5. **待機と復帰 (シェル)**: `ls` が仕事をしている間、シェルは待機しています。`ls` の処理が終わってプロセスが消滅すると、シェルが再び「次の入力を待つ状態」に戻ります。

### 例外：シェル組み込みコマンド (Built-in)
実は `cd` や `echo` のように、ディスク上のファイルではなく「シェル自身が内蔵している機能」として直接処理されるコマンドもあります。これを**ビルトインコマンド**と呼びます。しかし、基本的な関係性（シェルが解釈して処理を行う）は変わりません。

**まとめ**：
シェルは「入力文字を解釈し、他のプログラムを呼び出すための**親玉プログラム**」であり、コマンドはシェルによって呼び出される「**子分の実行ファイル**」です。ターミナルと人間を繋ぐ一番外側の皮がシェルで、その中身として個々のコマンドが詰まっているイメージです。
