# [[SSH]]の[[公開鍵]]認証の仕組みと「[[公開鍵]]の渡し方」

[[SSH]]（Secure Shell）通信における**[[秘密鍵]]（Private Key）**と**[[公開鍵]]（Public Key）**の仕組み、そして最大の疑問である「通信前にどうやって相手（サーバー）に[[公開鍵]]を渡しているのか？」について解説します。

## 1. [[秘密鍵]]と[[公開鍵]]のイメージ
まずは直感的なイメージとして、この2つを「南京錠」と「その鍵」に例えます。

* **[[公開鍵]] (Public Key)** = **「開いた状態の南京錠」** 🔒
  * 誰に渡しても安全です（暗号化やロックに使います）。
* **[[秘密鍵]] (Private Key)** = **「南京錠を開けるための本物の鍵」** 🔑
  * あなたのパソコンの奥深く（`~/.ssh/` など）に厳重に保管し、絶対に他人に渡してはいけません。

[[SSH]]通信（[[公開鍵]]認証）の基本的なイメージは、「サーバーの入り口に**あなたの南京錠（[[公開鍵]]）**をあらかじめ登録しておき、あなたが接続しに来た時に**手元の鍵（[[秘密鍵]]）**でガチャっと開けて入る」というものです。

## 2. 最大の疑問：どうやって最初に「南京錠（[[公開鍵]]）」を渡すのか？
「通信する前に持ってるっていうけど、どこで知るの？」という疑問は非常に鋭く、この仕組みの核心です。

結論から言うと、魔法のように自動で伝わるわけではなく、**「本格的な通信（[[公開鍵]]認証）を始める前に、別の手段で何としてでも[[公開鍵]]のデータをサーバーに置いてこなければならない（事前登録）」**のです。

具体的には、以下のような「泥臭い」方法で事前登録を行っています。

### パターンA：一時的に「パスワード」を使って入室し、置いてくる
まだ南京錠（[[公開鍵]]）がかかっていないサーバーに対して、最初は**パスワード認証**でログインします。
ログインできたら、サーバー上の指定のファイル（`~/.ssh/authorized_keys`）に、自分の[[公開鍵]]の中身（`ssh-rsa AAAAB3...` のような長い文字列）をテキストとしてコピペして保存します。これ以降はパスワードを無効化し、鍵だけで入れるようにします。
* ※ この「パスワードで入って[[公開鍵]]を追記する」という作業を全自動でやってくれる `ssh-copy-id` という便利なコマンドもあります。

### パターンB：Web画面（[[AWS]]や[[GitHub]]など）から登録する
[[AWS]]でサーバーを立ち上げる時や、[[GitHub]]に自分のPCを登録する時によく使われる方法です。
ブラウザの管理画面で、自分の[[公開鍵]]の文字列をペーストして登録します。すると、[[AWS]]などのシステム側が**サーバーを起動するついでに、裏側で自動的にサーバーの中に[[公開鍵]]を配置**してくれます。

### パターンC：管理者に南京錠だけ渡す
会社のサーバーなどでは、「自分のパソコンで[[公開鍵]]（南京錠）だけを作って、その文字列をチャットなどでインフラ管理者に送る」という運用があります。管理者がサーバーに南京錠をセットしてくれれば、次からあなたは[[秘密鍵]]でログインできるようになります。

## 3. なぜこんな面倒なことをするのか？
「最初パスワードで入れるなら、ずっとパスワードで良くない？」と思うかもしれません。
しかし、パスワードは「総当たり攻撃（総当たりでパスワードを試すボット）」によって突破されるリスクが常にあります。

一方は数学的に不可能（宇宙が何度終わっても計算しきれない）です。そのため、**最初の1回だけ面倒な登録作業をしてでも、パスワードログインを禁止して[[公開鍵]]認証に切り替えるのが現代のサーバーセキュリティの常識**となっています。

## 4. 各パターンの「ゴリ押し感」の正体と実際の利用シーン
「無理やりじゃない？」「ゴリ押しでは？」という感覚は非常に正常です。実際、最初の1回目は**「別の手段を使って物理的・強制的にねじ込む」**というゴリ押しをしているのが正解です。それぞれの実際の利用シーンを深掘りします。

### パターンAの詳細：「自分でサーバーを構築する時」のド定番
VPS（さくらのVPS、ConoHaなど）を契約したり、自宅で[[Raspberry Pi]]を立ち上げた時に最もよくやる**超一般的な手法**です。
1. サーバー契約直後は、業者から「初期パスワード」が発行されます。
2. そのパスワードを使って**1回だけ**ログインし、自分の[[公開鍵]]（南京錠）をサーバー内に配置します。
3. 配置し終わったら、サーバーの設定ファイル（`sshd_config`）を書き換えて、**「今後、パスワードを使ったログインは一切禁止（PasswordAuthentication no）」**と設定します。
これで、世界であなたしか入れない強固なサーバーが完成します。最初の1回はパスワードに頼るしかないという「鶏と卵」を解決するための強行突破です。

### パターンBの詳細：[[GitHub]]であなたがすでにやっていること
「[[Git]]/[[GitHub]]を使っている」とのことですが、実はすでにこのパターンBを無意識にやっているはずです！
1. [[GitHub]]でリポジトリをクローンする際、[[HTTPS]]ではなく[[SSH]]（`git@github.com:...`）を使おうとするとエラーになります。
2. これを解決するために、自分のPCで `ssh-keygen` コマンドを叩いて鍵のペアを作った記憶はないでしょうか？
3. そして、生成された `~/.ssh/id_rsa.pub`（[[公開鍵]]）の中身をコピーして、**ブラウザで[[GitHub]]の「Settings > [[SSH]] and GPG keys」の画面に貼り付けて保存**したはずです。
これがまさに「Web画面から南京錠を渡す」行為です。あなたが手動でコピペしたその南京錠を使って、[[GitHub]]のサーバーは「今プッシュしてきたのは本当に本人か？」を[[秘密鍵]]で確認しているのです。

### パターンCの詳細：企業の「パスワード禁止サーバー」での鉄則
これが一番ゴリ押しに見えますが、**企業などのチーム開発では最も安全で一般的な手法**です。
企業の本番サーバーは、最初から「パスワードログイン完全禁止」で構築されています。つまり、パターンAの手法は使えません。
では新入社員はどうやってサーバーに入るのか？
1. 新入社員は自分のPCで南京錠（[[公開鍵]]）だけを作ります。
2. その[[公開鍵]]のファイルを、Slackや社内チケット（Jiraなど）でインフラ管理者にポンッと渡します。
3. すでにそのサーバーに入れる特権（root等）を持っている管理者が、新入社員の南京錠をサーバーのドアに追加します。
4. 管理者から「設定したよ」と連絡が来たら、新入社員はパスワードを知らなくてもいきなり[[SSH]]接続できるようになります。
「通信以外のルート（チャットやメール）」を使って南京錠を渡すという、究極のアナログ＆ゴリ押し手法ですが、パスワードが一切ネットワーク上を流れないため極めて安全なのです。
