# ブラウザとサーバーの情報のやり取り（HTTPとルーティング）

「ブラウザはHTMLとCSSの塊で、アルゴリズムのロジックなんてないのに、どうやってC#のプログラムに命令を出しているのか？」

これは**Webの仕組みのど真ん中を突く、最も素晴らしい疑問**です！
おっしゃる通り、ブラウザは「C#の関数を実行しろ」というような高度な命令は一切出せません。

ではどうしているのか？ 答えは**「単なるテキスト（文字列）の手紙（HTTPリクエスト）をサーバーに送りつけているだけ」**です。

## ブラウザが送っている「手紙」の正体

ブラウザは、ユーザーの操作（リンクをクリックする、ボタンを押す）に合わせて、決められたフォーマットの「テキストメッセージ」を作ってサーバーに送信します。これを**HTTPリクエスト**と呼びます。

### パターン1: リンクをクリックした時 (GETリクエスト)
ユーザーが `<a href="/User/Profile/5">プロフィールを見る</a>` というリンクをクリックした時、ブラウザは以下のような手紙をサーバーに送ります。

```text
（ブラウザからサーバーへの手紙の中身）
GET /User/Profile/5 HTTP/1.1
Host: www.example.com
```
*   **GET**: 「データをちょうだい」という意味。
*   **`/User/Profile/5`**: 「このURLのやつをね」という意味。

### パターン2: フォームの送信ボタンを押した時 (POSTリクエスト)
ユーザーが入力フォームに名前を入れて「登録」ボタンを押した時、ブラウザは以下のような手紙を送ります。

```html
<!-- HTMLのフォーム -->
<form action="/User/Register" method="POST">
  <input type="text" name="UserName" value="Yamada">
  <button type="submit">登録</button>
</form>
```
↓
```text
（ブラウザからサーバーへの手紙の中身）
POST /User/Register HTTP/1.1
Host: www.example.com

UserName=Yamada
```
*   **POST**: 「このデータを受け取って処理して」という意味。
*   **UserName=Yamada**: ブラウザが勝手にHTMLの `name` 属性と入力された値をくっつけて送ってくれます。

## サーバー側の「翻訳家」（ルーティング）

ブラウザの仕事は「テキストの手紙を送る」ところで完全に終わります。
では、この単なる文字列が、どうやってC#のプログラムに変わるのでしょうか？

ここで登場するのが、ASP.NET Core MVCの**「ルーティング（Routing）」**という超重要機能です。
ルーティングは、フレームワークの入り口に立っている**翻訳家（または受付係）**です。

1.  **受付**: サーバーに `GET /User/Profile/5` という文字列が届く。
2.  **翻訳 (ルーティング)**:
    *   受付係「`/User` だから、`UserController` クラスの担当だな」
    *   受付係「`/Profile` だから、その中の `Profile` メソッドを呼べばいいな」
    *   受付係「`/5` がついてるから、メソッドの引数（カッコの中）に `5` を渡してあげよう」
3.  **実行**: フレームワークが、あなたの書いたC#のコードを呼び出します。
    *   `UserController.Profile(5)` が裏で実行される。

## まとめ：コントローラーの本当の姿

おっしゃる通り、ブラウザ自体には一切のロジックがありません。ブラウザは「ただの注文書（URLとデータだけのテキスト）」をサーバーに投げつけるだけです。

その「テキストの注文書」を読み解き、**「C#のどのクラスの、どのメソッドを、どんな引数で実行するか」に変換して呼び出してくれる魔法の仕組み**こそが、「Webフレームワーク（MVC）」の正体です。

だからこそ、Controllerのメソッドは、URLの形とぴったり一致するようにルール（おまじない）に従って書く必要があるのです。
