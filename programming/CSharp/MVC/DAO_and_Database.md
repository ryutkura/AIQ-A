# DAO（Data Access Object）とMVCアーキテクチャの関係

演習でDAOクラスを作成し、そこに「データベース接続」と「データの選別（SQLの実行など）」を任せていたのは、**ソフトウェア設計として大正解（めちゃくちゃ理にかなっている）**です！

まさにそれが「役割分担（関心の分離）」の究極の目的です。

## さらに解像度を上げた全体の構成図

前回の「Service層」の話に「DAO」を追加すると、プロの現場で使われている標準的な設計（レイヤードアーキテクチャ）が完成します。

```text
[ブラウザ] (お客さん)
   ↓↑
[Controller] (ウェイター：窓口)
   ・「〇〇さん、リクエストが来ましたよ」とServiceに伝えるだけ。
   ↓↑
[Service] (料理長：ビジネスロジック)
   ・「この年齢なら割引を適用しよう」「このデータは不正だからエラーにしよう」といったルールの判断。
   ・自分ではDBを見に行かず、DAOにデータを持ってくるよう指示する。
   ↓↑
[DAO / Repository] (買い出し担当・冷蔵庫の管理人：データアクセスロジック)
   ・SQLを組み立てる、DBに接続する、データを取得・保存する「だけ」を行う。
   ・「割引」などのビジネスルールは一切知らない。
   ↓↑
[データベース] (冷蔵庫・倉庫)
```

## なぜDAOを作ったのか？（DAOの絶大なメリット）

演習でわざわざDAOを作らされたのには、以下の明確な理由があります。

### 1. SQL（データベースの都合）を他の場所に漏らさないため
もしControllerやServiceの中に直接 `SELECT * FROM Users WHERE ...` といったSQLを書いてしまうと、後から「データベースをMySQLからSQL Serverに変えよう」となったときに、あらゆるファイルを書き直す羽目になります。
DAOがあれば、**「変更するのはDAOクラスの中身だけ」**で済みます。

### 2. 「データの取得」と「計算ロジック」を分離するため
Service（料理長）は「取得したデータをどう処理するか」に集中したいのです。「どうやってデータベースに接続してデータを持ってくるか」はDAOに任せることで、Serviceのコードが劇的に読みやすくなります。

## 現代のC# (ASP.NET Core) におけるDAO

演習でDAOを学んだのは大正解ですが、今後ASP.NET Core MVCを学ぶ上で知っておくべき用語のアップデートがあります。

現代のC#（特にEntity Framework Coreを使っている場合）では、「DAO」という名前ではなく、**「Repository（リポジトリ）パターン」**と呼ばれたり、そもそも**「DbContext / DbSet」**という機能がDAOの役割を自動的に果たしてくれるようになっています。

*   昔: `UserDao.FindById(1)` のようにSQLを書いて自作していた。
*   今: `_dbContext.Users.FirstOrDefault(u => u.Id == 1)` のようにC#の構文（LINQ）を書くだけで、裏でフレームワークがDAOの役割をやってくれる。

**結論:**
演習での「DAOにDB接続を任せる」という経験は、フレームワークの設計思想を理解するための完璧な土台です！その「役割分担の感覚」を持っていれば、ASP.NET Core MVCの学習は一気にスムーズになります。
