# [[DAO]]（Data Access Object）と[[MVC]][[アーキテクチャ]]の関係

演習で[[DAO]][[クラス]]を作成し、そこに「[[データベース]]接続」と「データの選別（[[SQL]]の実行など）」を任せていたのは、**ソフトウェア[[設計]]として大正解（めちゃくちゃ理にかなっている）**です！

まさにそれが「役割分担（関心の分離）」の究極の目的です。

## さらに解像度を上げた全体の構成図

前回の「Service層」の話に「[[DAO]]」を追加すると、プロの現場で使われている標準的な[[設計]]（レイヤード[[アーキテクチャ]]）が完成します。

```text
[ブラウザ] (お客さん)
   ↓↑
[Controller] (ウェイター：窓口)
   ・「〇〇さん、リクエストが来ましたよ」とServiceに伝えるだけ。
   ↓↑
[Service] (料理長：ビジネスロジック)
   ・「この年齢なら割引を適用しよう」「このデータは不正だからエラーにしよう」といったルールの判断。
   ・自分ではDBを見に行かず、[[DAO]]にデータを持ってくるよう指示する。
   ↓↑
[DAO / Repository] (買い出し担当・冷蔵庫の管理人：データアクセスロジック)
   ・[[SQL]]を組み立てる、DBに接続する、データを取得・保存する「だけ」を行う。
   ・「割引」などのビジネスルールは一切知らない。
   ↓↑
[データベース] (冷蔵庫・倉庫)
```

## なぜ[[DAO]]を作ったのか？（[[DAO]]の絶大なメリット）

演習でわざわざ[[DAO]]を作らされたのには、以下の明確な理由があります。

### 1. [[SQL]]（[[データベース]]の都合）を他の場所に漏らさないため
もし[[Controller]]やServiceの中に直接 `SELECT * FROM Users WHERE ...` といった[[SQL]]を書いてしまうと、後から「[[データベース]]をMy[[SQL]]から[[SQL]] Serverに変えよう」となったときに、あらゆるファイルを書き直す羽目になります。
[[DAO]]があれば、**「変更するのは[[DAO]][[クラス]]の中身だけ」**で済みます。

### 2. 「データの取得」と「計算ロジック」を分離するため
Service（料理長）は「取得したデータをどう処理するか」に集中したいのです。「どうやって[[データベース]]に接続してデータを持ってくるか」は[[DAO]]に任せることで、Serviceのコードが劇的に読みやすくなります。

## 現代の[[C#]] (ASP.NET Core) における[[DAO]]

演習で[[DAO]]を学んだのは大正解ですが、今後ASP.NET Core [[MVC]]を学ぶ上で知っておくべき用語のアップデートがあります。

現代の[[C#]]（特にEntity Framework Coreを使っている場合）では、「[[DAO]]」という名前ではなく、**「Repository（リポジトリ）パターン」**と呼ばれたり、そもそも**「DbContext / DbSet」**という機能が[[DAO]]の役割を自動的に果たしてくれるようになっています。

*   昔: `UserDao.FindById(1)` のように[[SQL]]を書いて自作していた。
*   今: `_dbContext.Users.FirstOrDefault(u => u.Id == 1)` のように[[C#]]の構文（LINQ）を書くだけで、裏でフレームワークが[[DAO]]の役割をやってくれる。

**結論:**
演習での「[[DAO]]にDB接続を任せる」という経験は、フレームワークの[[設計]]思想を理解するための完璧な土台です！その「役割分担の感覚」を持っていれば、ASP.NET Core [[MVC]]の学習は一気にスムーズになります。
