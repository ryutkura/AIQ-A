# フレームワークのルールと[[C#]]の文法、独自ロジックの見分け方

フレームワークを学び始めた人が最も苦労する「ごちゃごちゃになる問題」に対する処方箋です。
この3つを頭の中で切り離すことができると、フレームワークの学習は一気に楽になります。

## 3つの要素の分類

コードを書くとき、あなたが書いているものは以下の3つのどれかに分類されます。

1. **[[C#]] 言語そのものの文法** (どこでも使える共通ルール)
2. **フレームワークの決まり事** (ASP.NET Core [[MVC]]特有の「おまじない」やルール)
3. **あなたのオリジナルロジック** (ビジネスロジック、やりたいことの本質)

### 具体的なコードでの見分け方

[[コントローラー]]のコードを例に、どれがどれに該当するかを分解してみましょう。

```csharp
using Microsoft.AspNetCore.Mvc; // ← [フレームワーク] [[MVC]]の機能を使うための宣言
using System.Collections.Generic; // ← [C#] 標準機能（リストなど）を使うための宣言

namespace PracticeApp.[[Controller]]s
{
    // [フレームワークのルール]: [[Controller]]という名前で終わり、[[Controller]]クラスを継承する
    public class User[[Controller]] : [[Controller]] 
    {
        // [フレームワークのルール]: Webからのアクセスを受け付ける窓口（アクションメソッド）
        // 戻り値は「IActionResult」にするという決まり。
        public IActionResult Profile(int id) 
        {
            // ==========================================
            // ↓↓↓ ここから [あなたのオリジナルロジック] ↓↓↓
            // ==========================================
            
            // [C#の文法]: 変数の宣言とif文、クラスのインスタンス化
            string message = "一般ユーザー";
            
            if (id == 1) 
            {
                message = "管理者";
            }
            
            var user = new UserData { Name = "テスト太郎", Role = message };
            
            // ==========================================
            // ↑↑↑ ここまで [あなたのオリジナルロジック] ↑↑↑
            // ==========================================

            // [フレームワークのルール]: 最後に[[View]]()メソッドにデータを渡して終了する
            return [[View]](user); 
        }
    }
}
```

## ごちゃごちゃを解消するための3つの考え方

### 1. フレームワークは「穴埋め問題」だと心得る
フレームワークとは、**「あらかじめ8割完成しているシステム」**です。
あなたは、フレームワークが用意した**「ここにあなたの処理を書いてね」という穴（メソッドの中身など）**を埋めているだけなのです。
- 外枠（クラスの継承、メソッドの戻り値の型など）は「フレームワークのルール」。
- 穴の中身（計算、条件分岐など）は「[[C#]]の文法」を使った「独自ロジック」。

### 2. 「コンソールアプリでも動くか？」を考える
そのコードがASP.NET [[MVC]]特有のものか、純粋な[[C#]]のロジックかを見分ける簡単なテストがあります。
**「このコードは、黒い画面のコンソールアプリ（main関数だけの世界）にコピペしても動くか？」**と考えてみてください。
- `if`文や`List`の操作、金額の計算などは動きます。＝ **[[C#]]の文法 / 独自ロジック**
- `IActionResult` や `return [[View]]();` は動きません。＝ **フレームワークのルール**

### 3. [[Controller]]を「ただの連絡係」にする（Fat [[Controller]]を避ける）
フレームワーク嫌いを加速させる最大の原因は、[[Controller]]の中に「複雑な計算」や「[[データベース]]接続」などの**独自ロジックを書きすぎてしまうこと**です。
[[Controller]]はあくまでウェイターです。複雑な料理（独自ロジック）は別の普通の[[C#]]クラス（[[Model]]やServiceクラスと呼ばれるもの）を作って、そこに書くようにしましょう。
そうすれば、「フレームワークと話すコード」と「純粋な[[C#]]の計算コード」が物理的（ファイル単位）に分離され、非常にすっきりします。
