# ASP.NET [[MVC]]の「[[モデル]]バインダー」超入門

「Route Values」「Query String」「Form Values」、全部同じに思える…という疑問、**大正解**です！
実は**「ブラウザからサーバーにデータを送る方法」という意味では、やっていることは全部同じ**です。

では、なぜ3種類もあるのか？ そして「[[モデル]]バインダー」とは何者なのか？
小学生でもわかる**「宅配便」の例え**で解説します。

---

## 1. [[モデル]]バインダー（[[Model]] Binder）の正体

[[モデル]]バインダーの正体は、サーバーの玄関に立っている**「超優秀な荷物仕分け係のおじさん」**です。

[[コントローラー]]（ウェイター）が `public IActionResult Profile(int id, string name)` というふうに、「`id`という数字と、`name`という文字が欲しい！」と構えているとします。

仕分け係のおじさん（[[モデル]]バインダー）は、ブラウザから届いた荷物の山の中から、**「idって書いてある荷物」と「nameって書いてある荷物」をあちこち探して回収し、綺麗に整えて[[コントローラー]]の手にポンッと渡してくれます。**

この「あちこち探す場所」が、以下の3つなのです。

---

## 2. データを送る3つの場所（荷物の届き方の違い）

「データを送る」という目的は同じですが、**「手紙のどこに書いて送るか」**が違います。

### ① Route Values（ルート値）：「宛名の住所そのもの」に含める
*   **見た目:** `http://example.com/User/Profile/5`
*   **特徴:** [[URL]]のパス（階層）の一部としてデータが埋め込まれています。
*   **宅配便で例えると:** 「東京都渋谷区5番地」のように、**宛先（住所）そのもの**にデータ（5）が含まれている状態。
*   **使い所:** 「そのデータがないと、どのページか特定できない」ような絶対に必須のデータ（ユーザーIDや商品IDなど）に使います。

### ② Query String（クエ[[リスト]]リング）：「封筒の表のメモ書き」
*   **見た目:** `http://example.com/User/Search?keyword=apple&sort=price`
*   **特徴:** [[URL]]の最後にある `?` の後ろにくっつける「おまけ」のデータです。
*   **宅配便で例えると:** 封筒の表に**「※りんご希望、安い順で」と付箋（メモ）を貼っている**状態。
*   **使い所:** 「検索キーワード」や「並び替え順」など、[[URL]]を共有（ブックマークやSNSでシェア）したい時に使います。付箋なので、なくてもページ自体は見られます。

### ③ Form Values（フォーム値）：「頑丈なダンボール箱の中身」
*   **見た目:** （[[URL]]には一切表示されません。裏側でこっそり送られます）
*   **特徴:** ユーザーが入力フォーム（`<form>`）に入力して「[[P[[OS]]T]]（送信）」したデータです。
*   **宅配便で例えると:** **中身が見えない頑丈なダンボール箱の中に、分厚い書類（データ）を入れて送る**状態。
*   **使い所:** パスワードなどの「人に見られたくない秘密のデータ」や、長文のブログ記事、画像ファイルなど「量が多いデータ」を送る時に使います。絶対に[[URL]]には出ません。

## 3. [[モデル]]バインダーの凄さ

もし[[モデル]]バインダーがいなかったら、プログラム（[[C#]]）でこんな面倒なことを書かなければいけません。

```csharp
// 【昔の面倒な書き方（[[モデル]]バインダーなし）】
public IActionResult Profile() 
{
    // 自分で全部の場所を探しに行かないといけない！
    string idStr = Request.RouteValues["id"];     // 住所に入ってるかな？
    if (idStr == null) {
        idStr = Request.Query["id"];              // じゃあ付箋に書いてあるかな？
    }
    if (idStr == null) {
        idStr = Request.Form["id"];               // じゃあダンボールの中かな？
    }
    
    int id = int.Parse(idStr); // さらに文字を数字に変換する...面倒！
}
```

しかし、**[[モデル]]バインダー（優秀な仕分け係）**がいるおかげで、こう書くだけで済みます。

```csharp
// 【今の書き方（[[モデル]]バインダーあり）】
public IActionResult Profile(int id) 
{
    // おしまい！
}
```

**[[モデル]]バインダーの仕事:**
「お、[[コントローラー]]君は `int`（数字）の `id` が欲しいんだな。よし、住所（Route）と、付箋（Query）と、ダンボール（Form）の中身を全部探して、見つけたら数字に変換して渡してやろう！」

### 結論
「やってることは同じじゃないか？」→ **大正解です。全部「ブラウザからサーバーへデータを送る方法」です。**
ただ、「[[URL]]（住所）に組み込むか」「[[URL]]のおまけ（付箋）にするか」「隠して（ダンボールで）送るか」という**用途・目的の違い**があるだけなのです。

## 4. なぜ全部[[P[[OS]]T]]（ダンボール箱）に統一しないのか？

「中身が見えなくて安全なら、全部[[P[[OS]]T]]（Form Values）に統一すればいいじゃないか」という疑問。これもエンジニアが一度は考える素晴らしい視点です！

しかし、全部[[P[[OS]]T]]にしてしまうと、**Webサイトとして致命的な不便さ**が生まれてしまいます。理由は大きく4つあります。

### 理由1：[[URL]]を友達にシェア・お気に入り登録できないから（最重要）
Amazonで「赤い靴」を「安い順」で検索したとします。
もし全部[[P[[OS]]T]]（隠しデータ）だったら、[[URL]]はただの `https://amazon.com/search` のままになります。
この[[URL]]を友達にLINEで送っても、友達が開くのは「検索前の空っぽの画面」です。なぜなら「赤い靴」「安い順」というデータが[[URL]]に含まれていないからです。
Query String（`?keyword=赤い靴&sort=安い`）や Route Values（`/item/12345`）があるからこそ、私たちは**「今見ている特定の画面」を[[URL]]だけで共有・保存できる**のです。

### 理由2：ブラウザの「戻る」「更新」ボタンがおかしくなるから
フォーム（[[P[[OS]]T]]）を送信した直後にブラウザの「更新（F5）」ボタンを押すと、「フォーム再送信の確認（このページを再読み込みしますか？）」という怖い警告が出た経験はありませんか？
全部[[P[[OS]]T]]に統一すると、ページを移動して「戻る」ボタンを押すたびにあの警告が出ることになり、ユーザー体験が最悪になります。

### 理由3：「[[P[[OS]]T]]だから盗聴されない」は大きな誤解だから
「[[URL]]に見えないから安全」というのは、実は**画面を見ている後ろの人から見えないだけ**です。
インターネット上の通信経路では、暗号化されていない限り、[[P[[OS]]T]]の中身（Form Values）も丸見えです。
現代は**[[HTTPS]]**という通信全体の暗号化が標準になっているため、[[URL]]（Route/Query）も[[P[[OS]]T]]の中身も**両方とも暗号化されて安全に運ばれます。**つまり「盗聴防止」の観点ではどちらを使っても同じなのです。

### 理由4：ルールの使い分け（見るだけ [[GET]] vs 変更する [[P[[OS]]T]]）
Webの世界（[[HTTP]]のルール）では、以下のような明確な住み分けがあります。
*   **[[GET]] (Route / Query):** データを**「見るだけ」**の時に使う。（何度繰り返しても安全）
*   **[[P[[OS]]T]] (Form):** データを**「追加・更新・削除する」**時に使う。（例：決済ボタン。2回押すと2回買ってしまうから危険）

**結論：**
「検索結果」や「プロフィール画面」のように**シェアしたい・見るだけのデータは [[URL]]（Route/Query）** に含め、「パスワード」や「購入ボタン」のように**隠したい・変更するデータは [[P[[OS]]T]]（Form）** を使う、という使い分けがWebの絶対ルールなのです。
