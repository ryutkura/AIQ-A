# 複雑なロジックはどこに書くべきか？（Fat Controller / Fat Model 問題）

「Controllerに書くな」「でもModelに書きすぎると保守性が終わる」。
これはMVCを学ぶ人が必ずぶつかる、非常に優れた疑問です。

結論から言うと、実務レベルのアプリケーションでは**「Service（サービス）クラス」**という新しい層（レイヤー）を作って、そこに複雑なロジックを書きます。

## 「Model」という言葉の罠

MVCにおける「Model」という言葉は、実は非常に曖昧で、役割が大きすぎるのが問題の原因です。
実際には、Modelは大きく2つの役割に分けるべきです。

1.  **データそのものを表すModel（Entity / DTO / ViewModel）**
    *   例：「ユーザーID」「名前」「メールアドレス」という**箱（入れ物）**。
    *   ここには**計算などのロジックは書きません。**ただデータを保持するだけです。
2.  **ビジネスロジックを実行するModel（Service / Logic）**
    *   例：「ユーザーが18歳以上か判定する」「カートの合計金額を計算して割引を適用する」。
    *   **複雑な処理はすべてここに書きます。**

「Modelに書きすぎると保守性が終わる」と言っている人は、「1」のデータの箱の中に、「2」の複雑な計算まで全部詰め込もうとしている（Fat Model）状態を指しています。

## 推奨されるファイル構成（Service層の導入）

MVCに「Service」という概念を足して、**「M-V-C-S」**のように考えるのが現代の主流です。

```text
/Controllers
    └── UserController.cs (ウェイター：依頼を受け取り、Serviceを呼び出し、Viewに渡すだけ)
/Models
    └── User.cs (食材：データベースのテーブルと1対1になるデータの箱。ロジックなし)
/Services (または /Logics, /UseCases など)
    └── UserService.cs (料理人：Controllerから呼ばれ、複雑な計算やDB保存を行う)
/Views
    └── ...
```

## 処理の流れ（改良版）

1.  **Controller:** リクエストを受け取る。
2.  **Controller:** `UserService.CalculateDiscount()` などを呼び出す。
3.  **Service (UserService):** Model（データ）を取り出し、複雑な計算（割引率など）を行う。
4.  **Service:** 計算結果をControllerに返す。
5.  **Controller:** 結果をViewに渡す。

## なぜ Service に分けるのか？（最大のメリット）

この `UserService.cs` に書かれたコードは、**「完全に純粋なC#のコード」**になります。
（ASP.NET Core MVCの `IActionResult` や `HttpContext` といったフレームワークの機能に一切依存しません。）

そのため、以下のような絶大なメリットがあります。

*   **テストが超簡単になる:** ブラウザやサーバーを起動しなくても、単純なC#の関数としてテスト（単体テスト）ができます。
*   **使い回しができる:** Web画面だけでなく、「スマホアプリ用のAPI」や「夜間に自動実行されるバッチ処理」からでも、同じ `UserService` を呼び出して使い回すことができます。（Controllerにロジックを書いてしまうと、これができません）。

**結論:** 「MVC」という言葉に縛られず、ControllerとModelの間に**「Service（ビジネスロジック専用のクラス）」**という層を新しく作って、複雑な処理はすべてそこに書きましょう。
