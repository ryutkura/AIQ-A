# センサーとボタン（異なる入力方式）の統合とアダプターパターン

## 質問
追加の要件として、センサー以外に「ボタンの押下」で入退場を記録しているゲートが既にあります。
ボタンのゲートは「機器の信号として0か1を[[Python]]サーバに送って入退場を判別する」仕組みで既に実装されています。
しかし今回のセンサーゲートは「2つのセンサーの反応の順番」で入退場を判定するため、既存の「0か1で判定するロジック」とどう統合すればいいか（ソフトウェア的にどう組めばいいか）迷っています。

## 回答
なるほど、すでに「0なら退場、1なら入場」という立派な既存システム（コアロジック）が動いているのですね！
であれば、その既存の仕組みは**一切変更しなくて大丈夫**です。

ソフトウェア[[設計]]の王道パターンである**「アダプターパターン（Adapter Pattern）」**という考え方を使えば、非常に綺麗に解決できます。

### 解決策のコンセプト：[[Python]]側で「翻訳係（アダプター）」を作る
海外旅行に行く時、コンセントの形が違うからといって家電そのものを改造したりしませんよね。「変換プラグ（アダプター）」を間に挟むはずです。

それと全く同じで、センサーから来る「順番による複雑な判定」を、[[Python]]の途中で**「あ、これは入場だから『1』だな」「退場だから『0』だな」と翻訳（変換）**してあげて、既存のロジックに流し込みます。

こうすることで、[[View]]や既存のプログラムからは「すべてボタンと同じように0か1が送られてきた」ように見えます。

### 具体的な実装イメージ
前回の `GateMonitor`（状態管理[[クラス]]）を、「翻訳係」として使います。

```python
# ---------------------------------------------------------
# 1. すでに動いている既存のロジック（ここは一切いじらない！）
# ---------------------------------------------------------
def process_entry_exit(gate_name, value):
    """
    既存の処理。valueが1なら入場、0なら退場として[[View]]に送る。
    """
    action = "入場" if value == 1 else "退場"
    print(f"[既存システム] {gate_name} で {action} が発生しました！ [[View]]に送信します。")
    # ... ここに[[View]]へ[[JSON]]を送る既存のコードがある ...


# ---------------------------------------------------------
# 2. [[Python]]サーバ側で持つマッピング（ボタンとセンサーを区別する）
# ---------------------------------------------------------
DEVICE_MAP = {
    # 既存のボタン式ゲート
    "MAC-BTN-1": {"type": "button", "gate_name": "Gate_A"},
    
    # 今回追加するセンサー式ゲート（内・外も定義）
    "MAC-SEN-1": {"type": "sensor", "gate_name": "Gate_B", "position": "outer"},
    "MAC-SEN-2": {"type": "sensor", "gate_name": "Gate_B", "position": "inner"}
}

# ---------------------------------------------------------
# 3. リクエストの受け口（ここで分岐して、最終的に同じ[[関数]]に流す）
# ---------------------------------------------------------
# ※ sensor_monitor は前回の `GateMonitor` のような判定[[クラス]]
sensor_monitor = GateMonitor() 

def handle_incoming_request(mac_address, payload_value):
    # 1. 端末の種類を特定する
    device_info = DEVICE_MAP.get(mac_address)
    if not device_info:
        return # 知らない[[MACアドレス]]は無視

    device_type = device_info["type"]
    gate_name = device_info["gate_name"]

    # 2. ボタンの場合は、そのまま既存ロジックに流す！
    if device_type == "button":
        # ボタンは payload_value に 0 か 1 が入ってきている前提
        process_entry_exit(gate_name, payload_value)

    # 3. センサーの場合は、翻訳係（モニター）を通して 0 か 1 に変換してから流す！
    elif device_type == "sensor":
        position = device_info["position"]
        
        # モニターに状態をチェックさせる（この中で時間差と順番を判定）
        result = sensor_monitor.check_sequence(gate_name, position)
        
        if result == "入場":
            # センサーが「入場」と判定したら、ボタンの「1」と同じように扱う！
            process_entry_exit(gate_name, 1)
        elif result == "退場":
            # センサーが「退場」と判定したら、ボタンの「0」と同じように扱う！
            process_entry_exit(gate_name, 0)
        else:
            # まだ片方しか反応していない場合は何もしない（待機）
            pass
```

### なぜこの形（アダプターパターン）が良いのか？
1. **既存システムが壊れない**：すでに動いている `process_entry_exit` というコアの処理を一切書き換えないため、バグを生むリスクがありません。
2. **センサー側は「0/1」を知らなくていい**：マイコン側で無理やり「0」や「1」のフラグを作って送信させる必要がなくなります。
3. **将来別のゲートが増えても安心**：例えば将来「顔認証ゲート（名前の文字列が送られてくる）」が増えても、翻訳係を作って「認証成功＝1」として既存ロジックに流し込むだけで簡単に統合できます。

「入力の種類（ボタンかセンサーか）」を入口で判定し、センサーの場合は順番判定をしてから **「0 か 1 に翻訳して既存の[[関数]]に渡す」**。この流れで組むと、非常にスッキリとソフトウェアを統合できます！
