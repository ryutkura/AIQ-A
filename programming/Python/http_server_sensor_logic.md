# センサーの順序による入退場判定とMAC[[アドレス]]のグルーピング

## 質問
マイコンが送信するのはセンサーが感知したかどうかの信号で、信号の受信の順番によって「入場」と「退場」を識別するシステムです。
この手のセンサー付きマイコンがかなり多くあり、更にMAC[[アドレス]]は既にゲート名に割り当てているので、どうやって1個のグループにするか迷っています。
[[JSON]]の内容を変更してグループの列を追加するのも手ですが、[[View]]の作業者の負担を増やしたくありません。
この場合、諦めて[[JSON]]にグループ列を追加すべきか、プログラム（[[Python]]側）で制御できるものなのか、提案と解説をお願いします。

## 回答
詳細な仕様が見えてきました！これは非常に面白く、そして**「[[Python]]サーバ側（バックエンド）でプログラム制御すべき」**典[[型]]的なケースです。

結論から言うと、**[[JSON]]の仕様変更（マイコン側の変更や、[[View]]側の負担増）は不要**です。[[Python]]側で賢く制御（吸収）するのが、システム全体の[[設計]]として最も美しいアプローチです。

マイコン（[[IoT]]機器）は「自分が感知した」という事実だけを送り、[[View]]（フロントエンド）は「誰がどこから入退場した」という結果だけを受け取る。その間の「複雑な判定」をすべて[[Python]]が引き受ける構成を提案します。

---

### 提案：[[Python]]側でのマッピングとステートマシン（状態管理）

MAC[[アドレス]]が既に「ゲート名」に割り当てられているのであれば、[[Python]]側に「どのMAC[[アドレス]]が、どのゲートの、どちら側（内側か外側か）のセンサーか」という**[[辞書]]（マッピング表）**を持たせることで、完全にグループ化できます。

#### 1. [[辞書]]によるマッピング（紐付け）のイメージ
```python
# サーバ側で持つ設定（[[データベース]]やConfigファイルで管理しても良いです）
SENSOR_MAP = {
    "MAC-001": {"gate_name": "Gate_A", "position": "outer"}, # ゲートAの外側センサー
    "MAC-002": {"gate_name": "Gate_A", "position": "inner"}, # ゲートAの内側センサー
    "MAC-003": {"gate_name": "Gate_B", "position": "outer"}, # ゲートBの外側センサー
    "MAC-004": {"gate_name": "Gate_B", "position": "inner"}  # ゲートBの内側センサー
}
```
このように定義しておけば、マイコンから「MAC-001」が送られてきた時点で、[[Python]]は「Gate_Aの外側が反応したな」と瞬時にグループ（Gate_A）を特定できます。

#### 2. 入退場の判定ロジック（順番の判定）
順番による判定を行うため、[[Python]]側のメモリ上で「どのゲートが、最後にいつ、どちらのセンサーが反応したか」を記憶しておきます（前回の回答の `Aggregator` [[クラス]]を応用します）。

* **入場パターン**: `outer` が反応し、数秒以内に `inner` が反応した。
* **退場パターン**: `inner` が反応し、数秒以内に `outer` が反応した。

これをプログラムに落とし込むと以下のようになります。

```python
import time

class GateMonitor:
    def __init__(self):
        # ゲートごとの状態を保存する[[辞書]]
        # 例: {"Gate_A": {"last_trigger": "outer", "time": 1680000000.0}}
        self.gate_states = {}
        
        # MAC[[アドレス]]からゲート情報へのマッピング
        self.sensor_map = {
            "MAC-001": {"gate_name": "Gate_A", "position": "outer"},
            "MAC-002": {"gate_name": "Gate_A", "position": "inner"},
        }
        
        # 反応の有効期限（例えば5秒以内に両方反応しないと無効にする）
        self.timeout_seconds = 5.0

    def receive_signal(self, mac_address):
        # 1. 誰からの信号か特定する
        sensor_info = self.sensor_map.get(mac_address)
        if not sensor_info:
            return None # 登録されていないMACなら無視

        gate = sensor_info["gate_name"]
        position = sensor_info["position"] # "outer" or "inner"
        current_time = time.time()

        # 2. ゲートの現在の状態を確認する
        state = self.gate_states.get(gate)

        # 初めての反応、または前回の反応から時間が経ちすぎている場合
        if not state or (current_time - state["time"]) > self.timeout_seconds:
            # 状態を記録して、もう片方の反応を待つ
            self.gate_states[gate] = {"last_trigger": position, "time": current_time}
            return None # まだ判定できないので[[View]]には送らない

        # 3. すでに片方が反応していて、今回もう片方が反応した場合の判定
        last_position = state["last_trigger"]
        
        # 同じセンサーが2回連続で反応した場合は、時間を上書きして待つ
        if last_position == position:
            self.gate_states[gate]["time"] = current_time
            return None

        # 違うセンサーが時間内に反応した！ -> 入退場確定
        result_action = ""
        if last_position == "outer" and position == "inner":
            result_action = "入場" # 外 -> 内
        elif last_position == "inner" and position == "outer":
            result_action = "退場" # 内 -> 外

        # 状態をリセットする（次の人のために）
        self.gate_states[gate] = None

        # [[View]]に送るための確定データを返す
        return {
            "gate_name": gate,
            "action": result_action,
            "timestamp": current_time
        }
```

### なぜこの[[設計]]が良いのか？

1. **マイコンがシンプルになる**：マイコンは「自分が反応した！」というMAC[[アドレス]]を含んだ[[JSON]]を送るだけで済みます。グループ化のロジックを持つ必要がありません。
2. **[[View]]作業者が楽になる**：[[View]]側は、最終的に計算された `{"gate_name": "Gate_A", "action": "入場"}` という分かりやすいデータだけを受け取れば良いため、フロントエンドの実装が圧倒的に楽になります。
3. **拡張性が高い**：今後「Gate_C」が増えても、[[Python]]の `sensor_map`（設定）にMAC[[アドレス]]を2行追加するだけで対応完了します。

このように、**「センサーと[[ビュー]]の間に入って、複雑な翻訳と状態管理を行う」**のがバックエンド（[[Python]]）の醍醐味です。ぜひ諦めずに、[[Python]]側でこのロジックを組んでみてください！

## 追記・補足質問
**「いちいち[[JSON]]を参照せずに[[Python]]でマップしてしまうって事？もうすでにゲート名の割り当てでやってるけど、更にやった方がいいのかな。情報を増やしたり対応付けた方がいいのかもと思ってしまっている」**

### 補足への回答
言葉足らずで申し訳ありません、「すでにゲート名の割り当てをやっている」というのは素晴らしい状態です！まさに私が言いたかった「[[Python]]でのマッピング」はそれのことです。

もし現状で「[[MACアドレス]] → ゲート名」の変換ができているのであれば、**[[JSON]]にこれ以上無駄な情報を増やしたり、複雑な対応付けを追加する必要は全くありません。**

ただ1点だけ、「入場」か「退場」かを判定するためには、「同じゲートにある2つのセンサーのうち、どっちが『外側』で、どっちが『内側』か」を区別する必要がありますよね。
なので、現状の「ゲート名の割り当て設定」に、**「内側か・外側かの情報だけを1つ付け足す（紐付ける）」**だけでシステムは完成します。

* **今の状態**: `MAC-001 は ゲートA`
* **これだけでOK**: `MAC-001 は ゲートA の 外側`

この程度のマッピング情報（設定）を[[Python]]側に持たせておくだけで、[[JSON]]を複雑にすることなく入退場を完璧に捌くことができます！
