# ロバストネス図からシーケンス図への移植ガイドライン

**Q: ロバストネス図からシーケンス図への移植が上手くいきません。どの要素がどのアクターやオブジェクトになるのか、具体的な対応関係が分かりません。**

**A:**
ロバストネス図からシーケンス図への変換が難しく感じるのは、**「役割（抽象的な図）」から「実装（具体的な動き）」へと一気にステップアップするから**です。

まずは、それぞれの要素がシーケンス図のどこに対応するのかを整理しましょう。

### 1. 要素の対応表（マッピング）

| ロバストネス図の要素 | シーケンス図での役割 | 具体的なイメージ |
| :--- | :--- | :--- |
| **アクター** | **アクター** | システムを利用する「人間」や「外部システム」。一番左に配置。 |
| **バウンダリ** | **UIオブジェクト / API境界** | 画面（HTML）、ボタン、ダイアログ、外部連携インターフェース。 |
| **コントロール** | **メソッド / ロジッククラス** | 処理そのもの。シーケンス図では「矢印（メッセージ）」や「コントローラクラス」になる。 |
| **[[エンティティ]]** | **ドメインオブジェクト** | [[データベース]]に保存されるデータや, メモリ上のオブジェクト。 |

---

### 2. 移植の具体的な 3 ステップ

#### Step 1: 登場人物（ライフライン）を並べる
ロバストネス図に登場する要素を、シーケンス図の上部に「ライフライン」として並べます。

1.  左端に **アクター** を配置。
2.  その隣に、アクターが直接触れる **バウンダリ**（画面など）を配置。
3.  中央に **コントロール**（処理の司令塔）を配置。
4.  右側に **[[エンティティ]]**（データ）を配置。

#### Step 2: コントロールを「メッセージ（矢印）」に変える
ロバストネス図の「コントロール」は、シーケンス図では多くの場合、**バウンダリから送信されるメッセージ（メソッド呼び出し）**になります。

*   **ロバストネス図**: `[アクター] → [画面(B)] → [登録する(C)] → [注文(E)]`
*   **シーケンス図**:
    1.  アクターが「登録ボタン」を押す。
    2.  「画面オブジェクト」が「注文登録コントローラ」の `registerOrder()` メソッドを呼ぶ。**（←これがコントロールの正体）**
    3.  「コントローラ」が「注文[[エンティティ]]」を作成・保存する。

#### Step 3: 「何(What)」を「どうやって(How)」に変換する
ロバストネス図で「登録する」というコントロールがあったなら、シーケンス図では具体的に：
*   メソッド名は何か？（例：`save()`）
*   引数は何か？（例：`orderData`）
*   戻り値は何か？（例：`orderId`）
を考えて書き込みます。

---

### 3. なぜアクターすら分からなくなるのか？

「どのアクターになるか分からない」原因の多くは、**「システムの境界」が曖昧なこと**にあります。

*   **ヒント1: 誰がその処理をキック（開始）するか？**
    ロバストネス図で一番左側にいたアクターが、そのままシーケンス図の開始点になります。
*   **ヒント2: 外部システムはいないか？**
    「決済完了通知を受け取る」といった処理の場合、アクターは人間ではなく「決済代行システム」になります。

### まとめ：迷った時の合言葉
「ロバストネス図は**名詞（アクター、バウンダリ、[[エンティティ]]）**と**動詞（コントロール）**を見つけるための図。シーケンス図は、その**名詞の間で動詞をどうやり取りするか（メッセージ）**を描く図」

ロバストネス図のコントロールが「複数のメソッド」に分割されることもあります。まずは「役割」を「ライフライン」にそのまま置き換えるところから始めてみてください。
