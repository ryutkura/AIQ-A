# ヒアリング延期時の事前準備と仮説の立て方

**Q: ヒアリングがリスケになり、一日時間が空いてしまいました。提案の準備をしたいのですが、決め打ちしすぎてコミュニケーションが拗れるのは避けたいです。どの程度、何を準備するのが適切でしょうか？（ユースケース図や基本要求一覧は作るべき？）**

**A:**
一日時間が空いたのであれば、**「正解」を作るのではなく「仮説」を立てる**ことに時間を使うのが、最もリスクが低く、かつヒアリングの質を劇的に高める方法です。

「決め打ち」に見えないようにするためのコツは、**「当日はこれで行きましょう」ではなく、「現時点で理解できている範囲を整理したので、合っているか教えてください」というスタンス**で準備することです。

具体的に準備すべき内容と、その粒度を提案します。

---

### 1. 「基本要求一覧」：粒度は「疑問符付き」で
基本要求一覧は作るべきですが、確定事項としてではなく**「TODOリスト兼・確認事項リスト」**として作ります。

*   **やるべきこと**: 既に手元にある資料から、「システムができるべきこと（機能要求）」と「性能やセキュリティなどの条件（非機能要求）」を書き出す。
*   **「ほどほど」にするコツ**: 
    *   わからない部分を無理に埋めず、**「ここは要確認」「XXという解釈で合っているか？」というメモをそのまま残しておく。**
    *   顧客には「私たちの宿題リストとして整理しました」と見せることで、押し付け感を消せます。

### 2. 「ユースケース図」：粒度は「主要なものだけ」
ユースケース図は、**「システムの境界（どこまでがシステム化の対象か）」を合意するツール**として非常に優秀です。

*   **やるべきこと**: 「誰が（アクター）」「何を（ユースケース）」するのか、主要なハッピーパス（基本の流れ）を数個描いてみる。
*   **「ほどほど」にするコツ**:
    *   代替フロー（エラー系）や細かい分岐は描かない（これらはヒアリングで深掘りするネタにする）。
    *   「他にもこんなアクターはいませんか？」「この人はこの操作もしますか？」という**「問い」をセットで準備する。**

### 3. 「ドメイン用語集（暫定版）」：実はこれが一番おすすめ
一日時間があるなら、**用語集**を作るのが最も効果的です。

*   **やるべきこと**: 顧客の資料に出てくる「商品」「注文」「ステータス」などの言葉を抜き出し、自分なりの定義を書いてみる。
*   **メリット**: 言葉の定義がズレていると、後の設計がすべて崩れます。「この言葉ってこういう意味で使っていますか？」と確認する準備をしておくと、顧客は「自分たちの業務を理解しようとしてくれている」と信頼してくれます。

### 4. 「UIモックアップ（手書きレベル）」：論点整理のために
可能であれば、主要な画面の「ラフ書き」を準備します。

*   **やるべきこと**: ツールを使わず、白い紙やホワイトボードツールに「ここにボタンがあって、ここに入力欄がある」程度の絵を描く。
*   **メリット**: 「文字」だけだと認識がズレますが、「絵」を見せると「いや、ここにはXXの情報も表示したいんだよね」と顧客の口が滑らかになります。

---

### まとめ：時間の使い方の優先順位
1.  **用語集の作成**（認識のズレを最小化する）
2.  **ユースケース図のラフ作成**（範囲を合意する）
3.  **確認事項（質問）リストの整理**（ヒアリングの時間を最大化する）
4.  （余裕があれば）**主要画面のラフモック**（議論を具体化する）

「決め打ちした答え」を持っていくのではなく、**「議論を始めるための最高の地図」**を準備するつもりで取り組んでみてください。
