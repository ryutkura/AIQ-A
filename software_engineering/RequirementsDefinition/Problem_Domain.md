# [[問題領域]]（ドメイン）とは何か

**Q: この分野（[[要件定義]]・設計）における「[[問題領域]]（ドメイン）」とは何を指すのでしょうか？**

**A:**
「[[問題領域]]（Problem Domain）」あるいは単に「ドメイン」とは、**システム化の対象となる「現実世界での困りごと」や「ビジネスのルール」が渦巻いている範囲**のことを指します。

ソフトウェア開発を「道具作り」に例えると、[[問題領域]]は**「その道具を使って解決したい、現場の状況そのもの」**です。

### 1. 「[[問題領域]]」と「[[解決領域]]」の違い
この概念を理解するには、対になる「[[解決領域]]」と比較するのが一番わかりやすいです。

*   **[[問題領域]]（Problem Domain）**:
    *   **対象**: 顧客の業務、現行の商習慣、法規制、ユーザーの不満など。
    *   **関心事**: 「何に困っているか？」「どんなルールでビジネスが動いているか？」。
    *   **登場人物**: ユーザー、顧客、業務エキスパート（ドメインエキスパート）。
*   **[[解決領域]]（Solution Domain）**:
    *   **対象**: プログラミング言語、[[データベース]]、フレームワーク、AWS、デザインパターンなど。
    *   **関心事**: 「どうやって作るか？」「性能は出るか？」「保守しやすいか？」。
    *   **登場人物**: エンジニア、デザイナー、アーキテクト。

### 2. [[要件定義]]・設計においてなぜ重要なのか？
[[要件定義]]や[[ロバストネス分析]]で私たちがやっているのは、**「[[問題領域]]の混沌としたルールを、整理して[[解決領域]]（プログラム）へ橋渡しすること」**です。

*   **[[問題領域]]を理解せずに作ると？**:
    「コードは綺麗でバグもないが、現場の業務ルールに合っていないので誰も使わないシステム」が出来上がります。
*   **ドメイン[[モデル]]（[[エンティティ]]）の役割**:
    [[ロバストネス分析]]で抽出する「[[エンティティ]]」は、まさに[[問題領域]]にある「重要な概念（注文、在庫、会員など）」を抜き出したものです。

### 3. 具体例：図書館システムの場合
*   **[[問題領域]]**:
    *   「本を2週間以上借りるには特別な許可がいる」という業務ルール。
    *   「返却期限を過ぎると、新しい本は借りられない」という制約。
    *   「そもそも『貸出』とは、本と利用者と日付の組み合わせである」という概念。
*   **[[解決領域]]**:
    *   MySQLの `loans` テーブル。
    *   [[Java]]の `LoanService` クラス。
    *   Reactで作った返却ボタン。

### まとめ：設計者の役割
設計における「[[問題領域]]の把握」とは、**「プログラミングの話を一切せずに、その業務の専門家と対等に議論できるレベルで、相手のビジネスの仕組みを理解すること」**を指します。

[[要件定義]]の最初のステップは、この「[[問題領域]]」を正しく捉え、言葉の定義を揃えることから始まります。
