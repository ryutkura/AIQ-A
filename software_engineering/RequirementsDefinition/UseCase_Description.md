# [[ユースケース]]記述の実践的な書き方

**Q: [[ユースケース]]記述において、基本コースと代替コースの2つだけで1つの[[ユースケース]]としてまとめると参考書にはありますが、実務ではそれだけでは済まない（複雑すぎる）ケースがあります。そういった場合はどうすればよいのでしょうか？**

**A:**
おっしゃる通りです。参考書に載っている単純な「基本コース／代替コース」の枠組みだけで実際のシステム要件をすべて表現しようとすると、記述が肥大化して破綻（可読性の低下やメンテ不能）してしまいます。

実務で1つの[[ユースケース]]が複雑になりすぎた場合は、**「1つのテキストにすべてを詰め込むのをやめる」**というアプローチを取ります。具体的な対処法は以下の4つです。

### 1. [[ユースケース]]の分割（包含と拡張）
[[ユースケース]]が大きすぎる場合は、意味のある単位で別の[[ユースケース]]として切り出し、UMLの `<<include>>`（包含）や `<<extend>>`（拡張）を使って関係性を定義します。
*   **包含（include）**: 複数の[[ユースケース]]で共通して行われる処理を切り出す場合。（例：「商品を購入する」と「会員情報を変更する」の両方から、「ユーザー認証を行う」を切り出す）
*   **拡張（extend）**: 特定の条件下でのみ発生する、それ自体が独立した大きな処理を切り出す場合。（例：「商品を購入する」から、「ポイントを利用して購入する」を切り出す）

### 2. サブフローの活用
別の[[ユースケース]]として独立させるほどではないが、基本コースの中に書くと長すぎる場合、[[ユースケース]]記述の中に「サブフロー（内部フロー）」という項目を追加します。
*   **基本コースの記述**: 「3. システムは『パスワードリセットサブフロー』を実行する。」
*   **サブフローの記述**: その[[ユースケース]]の下部に、サブフロー専用の手順を記載する。
これにより、基本コースの「大まかな流れ」をスッキリ保つことができます。

### 3. 別ドキュメント（ビジネスルール）への委譲
特に代替コースが膨れ上がる原因の多くは、「細かいエラーチェック（バリデーション）」をすべて書こうとすることです。
*   **悪い例**: 「代替コースA: 名前が空の場合」「代替コースB: 名前が20文字以上の場合」「代替コースC: メール[[アドレス]]の形式が不正な場合」...
*   **良い例**: 「代替コースA: 入力値に不備がある場合、エラーメッセージを表示して手順2に戻る。※詳細な条件は『別紙：入力バリデーション定義表』を参照」
このように、細かい仕様はデータディクショナリやビジネスルール集といった別のドキュメントに逃がすのが鉄則です。

### 4. テキストを諦めて「図」を使う（アクティビティ図・状態遷移図）
[[ユースケース]]記述（テキスト）は、**「複雑な条件分岐」や「ループ（繰り返し）」を表現するのが非常に苦手**です。
テキストで「もしAならば手順3へ戻り、Bならば手順5へ進む。ただしCのフラグが立っている場合は…」と書き始めると、誰も読解できなくなります。
このような場合は無理に代替コースで表現せず、**「詳細な分岐フローについては、別紙のアクティビティ図（または状態遷移図）を参照」**として、図に役割を譲ります。

## 記述の粒度と注意点（Tips）

[[ユースケース]]記述を「いつ、どこまで書くか」についてのガイドラインです。

### 1. 適切な粒度：「ユーザーの目標（User Goal）」レベル
[[ユースケース]]の粒度は、アリスター・コーバーン氏が提唱した**「コーヒーブレイク・テスト」**で判断するのが最も実用的です。

*   **コーヒーブレイク・テストとは？**
    「その[[ユースケース]]が完了した直後、ユーザーは席を立ってコーヒーを飲みに行けるか？（＝一区切りついた満足感があるか？）」を問うテストです。

*   **判定例**:
    *   **失敗（細かすぎる）**: 「ログインする」
        *   ログインしただけでコーヒーを飲みに行ったら、同僚に「何しに来たの？」と言われます。これは目的（User Goal）ではなく、手段（Sub-function）です。
    *   **成功（適切）**: 「新規注文を登録する」
        *   注文を一件入れ終えたら、一仕事終えた実感が持てます。コーヒーを飲みに行っても不自然ではありません。
    *   **失敗（粗すぎる）**: 「在庫管理を行う」
        *   範囲が広すぎて、いつコーヒーを飲みに行けるか分かりません。これは複数の[[ユースケース]]をまとめた「サマリーレベル」です。

[[要件定義]]では、この**「コーヒーを飲みに行ける単位」**を一つの[[ユースケース]]として網羅することを目指します。

### 2. 書くときの3つの注意点（べからず集）

#### ① UIに依存しすぎない（実装非依存）
*   **NG**: 「OKボタンを押す」「プルダウンから選ぶ」
*   **OK**: 「入力を確定する」「項目を選択する」
    *   UI（ボタンかリンクか等）は後で変わる可能性があります。[[ユースケース]]記述では**「ユーザーの意図（何をしたいか）」**を記述し、具体的な操作方法は画面設計に委ねます。

#### ② 「中身（White-box）」を書かない
*   **NG**: 「システムはSQLを発行して在庫テーブルを更新し、ログを出力する」
*   **OK**: 「システムは在庫を確保し、処理の完了を記録する」
    *   システムを「ブラックボックス」として扱い、外部から見える振る舞いだけを書きます。内部設計をここに書くと、[[要件定義]]が技術仕様書になってしまいます。

#### ③ 受動態を避け、「主語＋述語」を明確にする
*   **NG**: 「データが送信される」
*   **OK**: 「ユーザーはデータを送信する」「システムはデータを受信する」
    *   「誰が何をするのか」を明確にしないと、責任分界点が曖昧になり、後の設計で「これどっちがやるんだっけ？」という混乱を招きます。

### まとめ
[[ユースケース]]記述は**「ユーザーが価値を得るための一連の対話」**を記録するものです。実装の細部に踏み込みすぎず、かといって抽象的になりすぎない「ユーザー目標」の視点を維持することが、良い[[要件定義]]のコツです。
