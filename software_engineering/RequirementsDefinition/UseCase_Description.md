# ユースケース記述の実践的な書き方

**Q: ユースケース記述において、基本コースと代替コースの2つだけで1つのユースケースとしてまとめると参考書にはありますが、実務ではそれだけでは済まない（複雑すぎる）ケースがあります。そういった場合はどうすればよいのでしょうか？**

**A:**
おっしゃる通りです。参考書に載っている単純な「基本コース／代替コース」の枠組みだけで実際のシステム要件をすべて表現しようとすると、記述が肥大化して破綻（可読性の低下やメンテ不能）してしまいます。

実務で1つのユースケースが複雑になりすぎた場合は、**「1つのテキストにすべてを詰め込むのをやめる」**というアプローチを取ります。具体的な対処法は以下の4つです。

### 1. ユースケースの分割（包含と拡張）
ユースケースが大きすぎる場合は、意味のある単位で別のユースケースとして切り出し、UMLの `<<include>>`（包含）や `<<extend>>`（拡張）を使って関係性を定義します。
*   **包含（include）**: 複数のユースケースで共通して行われる処理を切り出す場合。（例：「商品を購入する」と「会員情報を変更する」の両方から、「ユーザー認証を行う」を切り出す）
*   **拡張（extend）**: 特定の条件下でのみ発生する、それ自体が独立した大きな処理を切り出す場合。（例：「商品を購入する」から、「ポイントを利用して購入する」を切り出す）

### 2. サブフローの活用
別のユースケースとして独立させるほどではないが、基本コースの中に書くと長すぎる場合、ユースケース記述の中に「サブフロー（内部フロー）」という項目を追加します。
*   **基本コースの記述**: 「3. システムは『パスワードリセットサブフロー』を実行する。」
*   **サブフローの記述**: そのユースケースの下部に、サブフロー専用の手順を記載する。
これにより、基本コースの「大まかな流れ」をスッキリ保つことができます。

### 3. 別ドキュメント（ビジネスルール）への委譲
特に代替コースが膨れ上がる原因の多くは、「細かいエラーチェック（バリデーション）」をすべて書こうとすることです。
*   **悪い例**: 「代替コースA: 名前が空の場合」「代替コースB: 名前が20文字以上の場合」「代替コースC: メールアドレスの形式が不正な場合」...
*   **良い例**: 「代替コースA: 入力値に不備がある場合、エラーメッセージを表示して手順2に戻る。※詳細な条件は『別紙：入力バリデーション定義表』を参照」
このように、細かい仕様はデータディクショナリやビジネスルール集といった別のドキュメントに逃がすのが鉄則です。

### 4. テキストを諦めて「図」を使う（アクティビティ図・状態遷移図）
ユースケース記述（テキスト）は、**「複雑な条件分岐」や「ループ（繰り返し）」を表現するのが非常に苦手**です。
テキストで「もしAならば手順3へ戻り、Bならば手順5へ進む。ただしCのフラグが立っている場合は…」と書き始めると、誰も読解できなくなります。
このような場合は無理に代替コースで表現せず、**「詳細な分岐フローについては、別紙のアクティビティ図（または状態遷移図）を参照」**として、図に役割を譲ります。

### まとめ
参考書の「基本／代替コース」はあくまで基本形です。実務では**「ユースケース記述は全体の見通しを良くするための『目次』であり、複雑な詳細は別ファイルや図にリンクさせる」**という構成にすることで、破綻を防ぐことができます。
